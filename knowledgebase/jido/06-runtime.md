# 06 — Runtime: Jido Instance & AgentServer

> The **Jido instance** is your instance-scoped supervisor. The **AgentServer** is the GenServer that holds a single agent and drives the signal-directive loop.

## The Instance Module

```elixir
defmodule MyApp.Jido do
  use Jido, otp_app: :my_app
end
```

Optional `use` options:
- `:otp_app` (required) — your app name (e.g. `:my_app`)
- `:storage` (default `{Jido.Storage.ETS, [table: :jido_storage]}`) — default storage adapter
- `:default_plugins` (default `nil`) — override the framework-default plugin set

## Supervision Tree

```elixir
# lib/my_app/application.ex
children = [
  MyApp.Jido,
  MyApp.Repo,
  MyAppWeb.Endpoint,
  # InstanceManagers for durable agents/pods:
  Jido.Agent.InstanceManager.child_spec(name: :sessions, agent: MyApp.SessionAgent, ...),
  Jido.Agent.InstanceManager.child_spec(name: :pods, agent: MyApp.WorkspacePod, ...)
]
Supervisor.start_link(children, strategy: :one_for_one)
```

## What `use Jido` Supervises (per instance)

```
MyApp.Jido (Supervisor)
├── MyApp.Jido.TaskSupervisor    # Task.Supervisor, max_children: max_tasks (default 1000)
├── MyApp.Jido.Registry          # Registry, keys: :unique  (agent lookup by id)
├── MyApp.Jido.RuntimeStore      # ETS table for runtime relationships
└── MyApp.Jido.AgentSupervisor   # DynamicSupervisor, max_restarts: 1000, max_seconds: 5
```

Plus pool children if you configure `agent_pools:`.

## Instance API (auto-generated by `use Jido`)

```elixir
# Lifecycle
MyApp.Jido.start_agent(MyAgent, id: "a-1", initial_state: %{counter: 10})
MyApp.Jido.stop_agent("a-1")
MyApp.Jido.whereis("a-1")                      # => pid | nil
MyApp.Jido.list_agents()                       # => [{"a-1", pid}, ...]
MyApp.Jido.agent_count()                       # => integer

# Persistence
MyApp.Jido.hibernate(agent)                    # :ok | {:error, _}
MyApp.Jido.thaw(MyAgent, "a-1")                # {:ok, agent} | {:error, _}

# Debug
MyApp.Jido.debug()                             # current level
MyApp.Jido.debug(:on)                          # or :verbose, :off
MyApp.Jido.debug(pid)                          # toggle per-agent
MyApp.Jido.debug(:on, redact: false)           # disable redaction
MyApp.Jido.debug_status()                      # full status map
MyApp.Jido.recent(pid, 20)                     # ring buffer events

# Introspection
MyApp.Jido.registry_name()                     # MyApp.Jido.Registry
MyApp.Jido.agent_supervisor_name()
MyApp.Jido.task_supervisor_name()
MyApp.Jido.runtime_store_name()
MyApp.Jido.config(overrides \\ [])             # merged runtime config
```

All functions accept `partition: atom` keyword option for multi-tenancy ([12](12-multi-tenancy.md)).

## Top-Level `Jido.*` API

Same functions as the instance module, but take the Jido instance name as first arg (or a default `Jido.Default` instance for scripts/livebooks):

```elixir
Jido.start_link(name: MyApp.Jido)                          # start instance
Jido.start_agent(MyApp.Jido, MyAgent, id: "a-1")
Jido.stop_agent(MyApp.Jido, "a-1")
Jido.whereis(MyApp.Jido, "a-1")
Jido.list_agents(MyApp.Jido)
Jido.parent_binding(MyApp.Jido, child_id)                  # => {:ok, binding} | :error
Jido.hibernate(MyApp.Jido, agent)
Jido.thaw(MyApp.Jido, MyAgent, key)

# Quick-start (default instance Jido.Default)
Jido.start()                                               # idempotent
Jido.stop()                                                # stop default instance

# Await / coordination (delegated to Jido.Await)
Jido.await(server, timeout_ms)
Jido.await_child(server, tag, timeout_ms)
Jido.await_all([servers], timeout_ms)
Jido.await_any([servers], timeout_ms)
Jido.alive?(server)
Jido.get_children(parent)
Jido.get_child(parent, tag)
Jido.cancel(server, reason: :foo)

# Discovery (delegated to Jido.Discovery)
Jido.list_actions(name:, category:, tag:, limit:, offset:)
Jido.list_sensors(...)
Jido.list_plugins(...)
Jido.list_demos(...)
Jido.get_action_by_slug(slug)
Jido.refresh_discovery()
```

## Starting an Agent Directly (bypassing instance)

```elixir
{:ok, pid} = Jido.AgentServer.start_link(agent: MyAgent)
{:ok, pid} = Jido.AgentServer.start_link(agent: MyAgent, jido: MyApp.Jido, id: "a-1")
{:ok, pid} = Jido.AgentServer.start(agent: MyAgent, jido: MyApp.Jido)
```

You'll rarely do this outside tests — prefer the instance module.

Common `start_link/1` options:
- `:agent` (required) — agent module or pre-built struct
- `:jido` — instance name (for Registry/pools/storage)
- `:id` — string id, registered with the Jido Registry
- `:initial_state` — map merged into `agent.state` on start
- `:parent` — `%Jido.AgentServer.ParentRef{}` for child agents
- `:on_parent_death` — `:stop | :continue | :emit_orphan`
- `:debug` — `true | false | level` for per-agent debug
- `:error_policy` — see [19-errors.md](19-errors.md)

## Signal Processing Flow

```
Signal → AgentServer.call/cast
       → SignalRouter (strategy > agent > plugin routes by priority)
       → Agent.cmd/2
       → {updated_agent, directives}
       → Directives queued → drain via DirectiveExec
       → (for RunInstruction) execute → call cmd/2 again with result_action
```

## call/3 vs cast/2

```elixir
{:ok, agent} = Jido.AgentServer.call(pid, signal)
{:ok, agent} = Jido.AgentServer.call(pid, signal, 10_000)  # custom timeout
:ok = Jido.AgentServer.cast(pid, signal)
```

Default call timeout is 5000ms.

## Parent-Child Hierarchy

Jido's parent/child is a **logical hierarchy**, not nested OTP supervision.

- Parent and child agents are OTP peers under the same supervisor tree.
- Relationship tracked via `ParentRef`, child-start signals, process monitors.
- `on_parent_death` policies describe domain behavior, not OTP ancestry.

This is why a child can survive logical parent death and become orphaned without becoming an independently supervised OTP child of that parent. See [14-orphans-adoption.md](14-orphans-adoption.md).

### Parent receives (automatically wired)

- `jido.agent.child.started` — child is up
- `jido.agent.child.exit` — child exited (tracked children removed from `state.children` map)

### Child can emit to parent

```elixir
Directive.emit_to_parent(agent_or_ctx, signal)   # works only while attached
```

## Completion Detection

Agents signal completion via **state**, not process death. This lets you inspect results after.

```elixir
# Inside an action or hook
agent = put_in(agent.state.status, :completed)
agent = put_in(agent.state.last_answer, result)

# Externally
{:ok, state} = Jido.AgentServer.state(pid)
case state.agent.state.status do
  :completed -> state.agent.state.last_answer
  :failed    -> {:error, state.agent.state.error}
  _          -> :still_running
end
```

Status conventions: `:idle | :running | :waiting | :completed | :failed`. `Jido.await/2` understands these by default (customizable via `:status_path`, `:result_path`, `:error_path`).

## Debug & Diagnostics

- Instance-wide: `MyApp.Jido.debug(:on)` / `:verbose`
- Per-agent: `Jido.AgentServer.set_debug(pid, true)` or `start_agent(..., debug: true)`
- Ring buffer (up to 500 events, configurable): `MyApp.Jido.recent(pid, 20)` or `Jido.AgentServer.recent_events(pid, limit: 20)`

Events are newest-first with `:at`, `:type`, `:data` fields. See [18-debugging.md](18-debugging.md).

## Await / Timeout

```elixir
case Jido.await(pid, 5_000) do
  {:ok, result} -> result
  {:error, {:timeout, diag}} ->
    # diag: %{hint: ..., server_status: ..., queue_length: ..., iteration: ..., waited_ms: ...}
    Logger.warning("Agent await timed out", diag)
    {:error, :timeout}
end
```

See [16-worker-pools.md](16-worker-pools.md) for pool-backed call patterns and [20-testing.md](20-testing.md) for await patterns in tests.

## Choosing a Runtime Pattern

| Need | Use |
|---|---|
| One live agent | `MyApp.Jido.start_agent/2` (or `AgentServer.start_link/1`) |
| Live tracked child | `Directive.spawn_agent/3` |
| Durable named agent | `Jido.Agent.InstanceManager.get/3` ([10](10-persistence.md)) |
| Durable named team | `Jido.Pod.get/3` ([11](11-pods.md)) |
| Tenancy | add `partition:` ([12](12-multi-tenancy.md)) |
